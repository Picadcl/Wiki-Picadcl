# query

## 概览

使用这个 skill 基于已维护的 wiki 页面回答用户问题。遵守 `AGENTS.md` 中的项目规则：先从 `wiki/index.md` 建立主题视野，再进入相关主题目录阅读 `overview.md`、`_index.md` 和最相关页面。回答要明确依据来自哪些页面或来源；只有在用户要求归档时，才把回答沉淀到 `wiki/topics/<topic>/questions/`。

## 工作流

### 1. 识别问题范围

- 先阅读 `wiki/index.md`。
- 根据问题判断所属主题；如果不确定，使用 `rg` 搜索 `wiki/topics/`。
- 如果问题横跨多个主题，分别识别主要主题和辅助主题。
- 如果 wiki 中明显没有足够依据，直接说明缺口，不要用未记录的信息补齐结论。

### 2. 阅读相关上下文

- 优先阅读对应主题的 `overview.md`。
- 如果存在 `wiki/topics/<topic>/_index.md`，继续阅读主题内索引。
- 阅读最相关的来源摘要、概念页、实体页或问题页。
- 如需追溯原始材料，再读取对应 `raw/<topic>/` 文件。
- 对跨主题问题，阅读相关主题的入口页面，并注意不同主题语境下同名概念可能并不等同。

### 3. 形成回答

- 回答要区分“wiki 已有结论”“来源中的主张”和“基于页面之间关系作出的推断”。
- 对不确定、冲突或材料不足的部分，标记为开放问题。
- 尽量使用 Obsidian 风格页面名指代内部依据，例如 `[[概念名称]]`。
- 如果用户要求比较、决策或分析，说明比较维度和依据来源，避免只给单句判断。

### 4. 必要时归档问题

仅当用户明确要求归档，或用户要求“保存这个回答/整理成长期页面”时，才创建或更新 `wiki/topics/<topic>/questions/` 下的页面。

归档时：

- 优先更新已有相关问题页，而不是创建近似重复页面。
- 使用 `wiki/_templates/question.md` 作为结构参考。
- 文件名使用 kebab-case。
- frontmatter 包含 `title`、`type`、`created`、`updated`、`sources`、`tags` 和 `topic`。
- 正文保留本次回答的核心结论、依据页面、未解决问题、相关页面和后续可追踪线索。
- 添加到相关概念页、实体页、主题 `_index.md` 的链接。


## 最终回复

回答完成后，根据实际情况报告：

- 使用的主题。
- 主要依据页面或来源。
- 回答结论。
- 是否创建或更新了归档页面。
- 仍未解决的问题、矛盾或材料缺口。
